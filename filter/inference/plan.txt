SentinelAI Filter Inference Service - Implementation Plan
========================================================

GOAL: Production-ready gRPC inference service for SentiBERT model

---

PHASE 1: Protocol Definition & Code Generation
-----------------------------------------------
[ ] Define filter.proto with service contract:
    - ClassifyMessage RPC (single prediction)
    - ClassifyBatch RPC (batch predictions, optional)
    
[ ] Message definitions:
    - ClassifyRequest: { message: string, metadata?: map }
    - ClassifyResponse: { 
        category: string,
        category_confidence: float,
        severity: string, 
        severity_confidence: float,
        is_risk: bool,
        risk_score: float
      }
    - BatchRequest/BatchResponse for bulk operations

[ ] Generate Python code from proto:
    - filter_pb2.py (messages)
    - filter_pb2_grpc.py (service stubs)

---

PHASE 2: Core Inference Logic
------------------------------
[ ] Create inference/engine.py:
    - InferenceEngine class
    - Model loading with model_factory.load_production_model()
    - Singleton pattern for model caching (load once, reuse)
    - predict() method: raw text -> (category, severity, scores)
    - Batch predict support

[ ] Implement preprocessing:
    - Text normalization (if needed)
    - Tokenization using config.MODEL_NAME tokenizer
    - Batching logic

[ ] Implement postprocessing:
    - Logits -> probabilities (softmax)
    - Apply risk threshold from config (or ROC-tuned threshold)
    - Map indices back to category/severity labels

---

PHASE 3: gRPC Server Implementation
------------------------------------
[ ] Create inference/server.py:
    - FilterServiceServicer class (implements generated interface)
    - ClassifyMessage handler:
        * Validate input
        * Call InferenceEngine.predict()
        * Map to ClassifyResponse proto
        * Error handling (malformed input, model errors)
    
[ ] Server lifecycle:
    - Initialization: Load model on startup
    - Graceful shutdown: Cleanup resources
    - Health check endpoint (optional)

[ ] Configuration:
    - Port selection (e.g., 50051)
    - TLS/SSL setup (optional for internal microservices)
    - Max workers / thread pool settings

---

PHASE 4: Error Handling & Resilience
-------------------------------------
[ ] Handle error cases:
    - Empty/invalid messages -> gRPC INVALID_ARGUMENT
    - Model loading failures -> gRPC UNAVAILABLE
    - Inference timeouts -> gRPC DEADLINE_EXCEEDED
    - CUDA OOM / memory issues -> Fallback to CPU

[ ] Logging:
    - Request logging (message hash, not full text for privacy)
    - Inference latency tracking
    - Error tracking with stack traces

[ ] Monitoring hooks:
    - Request counter
    - Latency histogram (p50, p95, p99)
    - Error rate

---

PHASE 5: Testing & Validation
------------------------------
[ ] Unit tests (pytest):
    - Test InferenceEngine with real model
    - Test prediction correctness on known samples
    - Test batch processing

[ ] Integration tests:
    - Start server in test mode
    - Create gRPC client
    - Send test requests, validate responses
    - Test error scenarios

[ ] Performance benchmarking:
    - Measure single prediction latency
    - Measure throughput (requests/sec)
    - Stress test with concurrent requests

---

PHASE 6: Deployment Preparation
--------------------------------
[ ] Docker support:
    - Dockerfile for inference service
    - Include model weights or download on startup
    - Expose gRPC port

[ ] Docker Compose integration:
    - Add filter-inference service to docker-compose.yaml
    - Network configuration for microservice access
    - Health checks

[ ] Documentation:
    - Usage examples (Python client)
    - Client examples for other languages (Node.js stub)
    - API reference from proto

[ ] Client library (optional):
    - Wrapper class for easy consumption
    - Connection pooling
    - Retry logic

---
